Designing a centralized alert routing system using AWS Lambda

Beyond this article

Who wouldn’t love a centralized monitoring and alerting system, right? Are you in the process of consolidating all your alerting systems into a single one? Felt the pain of implementing alerting to a different channel for other stakeholders or teams? Don’t even want to talk about the hassle of switching from one provider to another, like Google Chat to Slack?

In this blog, I will explain how you can configure a centralized alert routing system using an AWS Lambda function as its core while making it easy to switch providers. Let me start with some requirements and then derive a solution from them.

Any centralized alerting system should consist of the following:

  1. Easy to switch alerting providers i.e. Slack, Google Chat, etc.
  2. Simple enough to adapt to multiple alerting providers in parallel depending on team/use case.
  3. Handling grouped alerts together, i.e., MySQL vs. RabbitMQ specific implementation.
  4. Environment-specific implementation, i.e., production, staging, etc.
  5. Extensible enough to call other providers like Opsgenie or simple webhook API calls etc.

The high-level design of the alerting system using AWS Lambda

I don’t think anybody can cover all the use cases in a single post. Hence I am going to briefly cover all the points mentioned above. Below, I am explaining the process in an abstract manner.

Input for Lambda: Input can be from multiple sources. For example, Amazon Simple Notification Service(SNS) topic, invocation from Amazon EventBridge rule, or direct invocation using Lambda SDK clients. To support external clients via http, you can also invoke it via Amazon API Gateway.

Transform & Notify: We need to generate a generic payload from the input and pass it to our final notifier function, where all the necessary channels or providers will be checked against the payload and delivers the notification appropriately.

Pseudo code for the Lambda function

Here, I am only considering two types of input sources. One is the AWS SNS topic, and another is via Amazon EventBridge invocation. But you can have any number of sources and their own implementation logic as per your needs.

lambda_handler(event:, context:)

  - Set environment variables required for further processing

End result

Once your unique set up is complete, the notifications will start to arrive on the messaging app of your choice. You can see Google Chat and Slack notifications in the gif below. I have generated dummy alerts to give you an idea of the end result generated via sample implementation below. I’ve given the same dummy alerts generator script in the GitHub gist mentioned as part of sample implementation.

Sample implementation using Ruby and Prometheus as alert source

FYI, I am not giving any steps to configure the Lambda function itself as AWS documentation covers it well. Please read it once and you will be able to set it up easily.

Here is my full GitHub gist link. For reference, I am only attaching main Lambda function’s ruby script here.

require 'time'
require 'json'
require './notify_gchat.rb'
require './notify_slack.rb'

def generate_alert_kv_pairs(payload)
  kv_pairs = []
  case payload['type']
  when 'custom_alert'
    # Handling your custom alert fields
    kv_pairs << ['Type', payload['condition']]
    kv_pairs << ['Description', payload['message']]
  when 'prometheus_alert'
    # Handle prometheus alert fields
    kv_pairs << ['Type', payload['condition']]
    kv_pairs << ['Status', payload['status']]
    kv_pairs << ['Summary', payload['summary']]
    kv_pairs << ['Description', payload['description']] if payload['description']
  when 'ecs_alert'
    kv_pairs << ['Type', payload['condition']]
    # Need to extract details from different values when payload is from aws eventbridge invocation
    if payload['instance']
      kv_pairs << ['Cluster', payload['cluster'].split('/').last]
      kv_pairs << ['Container Instance', payload['instance'].split('/').last]
    elsif payload['cluster']
      kv_pairs << ['Cluster', payload['cluster'].split('/').last]
      kv_pairs << ['Service', payload['service'].split('/').last] if payload['service']
    elsif payload['service']
      cluster_and_svc_name = payload['service'].sub(/[^\/]*\//, '').split('/')
      kv_pairs << ['Cluster', cluster_and_svc_name.first]
      kv_pairs << ['Service', cluster_and_svc_name.last]
    end
    # Add conditional fields if they are only required for some alerts
    kv_pairs << ['Reason', payload['reason'].sub(/[^\/]*\//, '')] if payload['reason']
  else
    # A fallback if type isn't defined
    kv_pairs << ['Unknown Alert Type', "Yes"]
    kv_pairs << ['Alert JSON', payload.to_json]
  end
  return kv_pairs
end

def skip_notification(payload)
  # skip notification as per your custom conditions
  return false
end

def notify(payload)
  @logger.info("Notification payload received: #{payload.to_json}")
  if skip_notification(payload)
    @logger.info("Notification has been skipped.")
    return
  end
  payload['timestamp'] = payload['timestamp'] ? Time.parse(payload['timestamp']).getlocal : Time.now
  # falling back to lambda env variable if an payload doesn't contain environment
  payload['environment'] = payload['environment'] || ENV['DEFAULT_ENV']
  # Falls back to default notification sender
  payload['send_to'] = ENV['DEFAULT_ALERT_SEND_TO'] if payload['send_to'].nil?
  # instead of just kv_pairs, you can enhance your code to include grafana / aws console links too
  kv_pairs = generate_alert_kv_pairs(payload)
  case payload['send_to']
  when 'slack'
    notify_slack(payload, kv_pairs)
  when 'google_chat'
    notify_gchat(payload, kv_pairs)
  else
    # Just in case of typo / user forgot to update field
    notify_slack(payload, kv_pairs)
  end
end

def update_payload_custom_logic(payload)
  title_status = "#{payload['environment']}/#{payload['status'].downcase}"
  source = payload['source']
  # custom logic implementation using source field
  if ! source.nil? && source != ""
    if source.start_with?('ecs/')
      title_status += "/#{source.split('/')[2]}"
    elsif source.start_with?('targetgroup/')
      title_status += "/#{source.split('/')[1]}"
    elsif source.start_with?('url_monitoring/')
      title_status += "/#{source.sub(/.*http[s]:\/\/(.*).alertrouter.com.*/, '\1')}"
    else
      title_status += "/#{source}"
    end
  end
  # custom logic trimmed to only update title for notification
  # you can update it to enhance it more by adding links to
  # grafana dashboards / aws console / runbooks / custom urls
  return payload.merge({'title' => "[#{title_status}] #{payload['condition']}"})
end

def handle_prometheus_alert(timestamp, alert)
  payload = {
    'type' => 'prometheus_alert',
    'send_to' => alert['labels']['send_to'],
    'severity' => alert['labels']['severity'],
    'condition' => alert['labels']['condition'],
    'timestamp' => timestamp,
    'environment' => alert['labels']['env'],
    'summary' => alert['annotations']['summary'],
    'status' => alert['status'].upcase,
    'description' => alert['annotations']['description'],
    'source' => alert['annotations']['source']
  }
  notify(update_payload_custom_logic(payload))
end

def handle_sns_message(message_json, timestamp)
  # please update parsing logic if you are not using json or handling multiple input types
  message = JSON.parse(message_json)
  if message['source'] == 'prometheus'
    message['alerts'].each { |alert| handle_prometheus_alert(timestamp, alert) }
  end
  # Handle other sources
end

# add exception handling in your own way
# this is a very trimmed version to give you an idea
# please feel free to update it to use for your own use cases
def lambda_handler(event:, context:)
  @logger = Logger.new($stdout)
  @logger.info("Event received: \n#{event.to_json}")
  # checking the source of the alert(s)
  if event['Records'] && event['Records'][0]['EventSource'] == 'aws:sns'
    @logger.info("Message from SNS")
    handle_sns_message(event['Records'][0]['Sns']['Message'], event['Records'][0]['Sns']['Timestamp'])
  else
    # direct lambda invocation
    environment = event['environment'] || ENV['DEFAULT_ENV']
    event['title'] = "[#{environment}] #{event['condition']}"
    notify(event)
  end
end

Beyond this article

I know I have only touched the surface and there is much more to be done. I want you to start small first and then implement a full scale solution for your organization. Some of you might think that given so many alerting tools, why we should reinvent the wheel again? The answer is simple. Customization. There is no one stop solution for all the use cases. But with a system like this, you can able to achieve higher customization for your organization easily.

My other article on AWS ECS Monitoring builds on concepts discussed in this blog. The info shared in that blog can be integrated with this setup and have your ECS cluster(s) monitored proactively without any third party agents running.
