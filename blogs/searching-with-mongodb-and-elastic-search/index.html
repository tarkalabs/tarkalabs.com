Searching Data

Nowadays searching has become a necessary part of any web or mobile application. Consider a case of an E-Commerce website with no searching capabilities, you might end up going to competitor sites to find your favourite headphones or if you are an admin of the same website and checking logs for errors, viewing millions of logs without search is as good as useless for you.

We can use different architectures for storing data and searching. Using MongoDB as datastore and Elastic search for Searching is a commonly used architecture. MongoDB provides a NoSQL DBMS for storing huge amount of data. Elastic search on the other hand provides capability to store, index, search and analyse data in real time to extract value from datastore.

This article is hands on and mostly focuses on basic to advanced level searching in a healthcare application. Let’s start with making basic Patient model to store data in MongoDB using NodeJS and mongoose.

import mongoose from "mongoose";
var Schema = mongoose.Schema;

var patientSchema = new Schema(
  {
    name: {
      type: String
    },
    email: {
      type: String
    },
    phone: {
      type: String
    },
    type: {
      type: String
    },
    prescriptions: {
      type: [String] // prescriptions is an array of String filepaths
    }
  },
  {
    timestamps: true
  }
);

module.exports = mongoose.model("Patient", patientSchema);

For simplicity we are considering the patient to contain name, type(Type of disease the patient has) and an array prescriptions containing local file paths where the prescription is stored as a file. We can perform following possible search operations on this model.

  • Search patients based on name or email.
  • Full text partial search based on name, email, phone or type of patient (Any field).
  • Full text partial search based on name, email, phone, type of patient or file search in prescription documents. (A possible use case for this could be to search for list of all patients who have been given a particular medicine in the last two weeks).

Solutions

Search patients based on patient name or email

This one is pretty straight forward and we can use mongoose queries to achieve this easily.

let patient = await Patient.find({
  name: name,
  email: email
)};

Full text partial search based on any field

This is when things start getting complicated and we introduce Elastic search to improve our search capabilities. We can still achieve this in MongoDB by making use of regular expressions and search using OR conditions in every field but elastic search can make our life much easier here.

We will use mongoosastic to integrate elastic search with MongoDB. Mongoosastic is a mongoose plugin which can automatically index your models into elastic search.

To achieve this, we need to first index the required fields using mongoosastic and then update our search query to use elastic search.

import mongoose from "mongoose";
import mongoosastic from "mongoosastic";
import elasticsearch from "elasticsearch";
import Organization from "./organization";

var Schema = mongoose.Schema;

var patientSchema = new Schema(
  {
    name: {
      type: String,
      es_indexed: true
    },
    email: {
      type: String,
      es_indexed: true
    },
    phone: {
      type: String,
      es_indexed: true
    },
    type: {
      type: String,
      es_indexed: true
    },
    org: {
      type: [Organization]
    },
    prescriptions: {
      type: [String]
    }
  },
  {
    timestamps: true
  }
);

patientSchema.plugin(mongoosastic, {
  index: `${process.env.NODE_ENV || "dev"}:patients`,
  esClient: elasticsearch.Client({
    host: elastic_search_host // localhost:9200 or your elastic search url
  }),
  populate: [
    {
      path: "org",
      // This takes an assumption that organization has
      // a name field which needs to be indexed.
      select: "name"
    }
  ]
});

module.exports = mongoose.model("Patient", patientSchema);

We have made a little complex Patient model here by adding another field Organization which is another Schema. In the above example name, email, phone, type and org.name are the indexed fields and we can search using these fields.

Now assuming if elastic search is running at localhost:9200(default elastic search url), the above index can be searched as

http://localhost:9200/dev:patients/_search

By default all the new Patients are saved into elastic search however if you have existing data which needs to be synchronised, we can use the following command for the same

await Patient.synchronize()

Once all the existing patients have been synchronised from MongoDB to Elastic search, we can use the below wildcard query to fetch and display results based on partial full text search

async function getAllPatients(search) {
  return new Promise((resolve, reject) => {
    Patient.search({
      query_string: {
        query: `*{search}*`,
        analyzer: "keyword"
      }
    },
    async (err, results) => {
      if (!results || !results.hits || results.hits.total <= 0) {
        return resolve([])
      }
      let patientIds = results.hits.hits.map(r => r['_id']);
      resolve(
        await Patient.find(_id: {
          $in: patientIds
        })
      )
    })
  })
}

This will allow the user to do a partial search on any of the indexed fields as we are using a wildcard operator * while searching. This technique works well for string based fields and can scale quite well. A new complex problem statement in this domain would be to search for patients based on a text in the prescription files (For this problem statement, we are considering prescriptions to be stored as text files in file storage and path of the same is stored as an array of String in Patient model).

Full Text Search on text fields and files

(Assumption: We will not get into specifics of how to search in a prescription file but consider it to be match if we have a partial text match in any prescription of a user)

Elasticsearch Plugins: Plugins are a way to enhance the basic Elastic search functionality in a custom manner. We will make use of Elastic search “Ingest Attachment Processor Plugin” for the problem at hand. The ingest attachment plugin lets Elastic search extract file attachments in common formats(such as PPT, XML, PDF) by using Apache text extraction library Tika.

The steps involved in this searching are

  • Creating a Pipeline
  • Indexing your data
  • Searching for your data

Creating a Pipeline

import request from "request-promise-native";

async function createPipeline() {
  let elastic_search_url = process.env.ES_URL || "localhost:9200";

  let requestOptions = {
    method: "PUT",
    headers: {
      "Content-Type": "application/json"
    },
    body: JSON.stringify({
      description: "Extract attachment information",
      processors: [
        {
          attachment: {
            field: "data"
          }
        },
        {
          remove: {
            field: "data"
          }
        }
      ]
    }),
    url: `http://${elastic_search_url}/_ingest/pipeline/attachment`
  };

  return await request(requestOptions);
}

Once the ingest pipeline is created, the next step is to index your files on a unique parameter.

Indexing Data

async function createIndex(id, data, type, createdAt) {
  let elastic_search_url = process.env.ES_URL || 'localhost:9200'

  let requestOptions = {
    method: 'PUT',
    headers: {
      'Content-Type': 'application/json'
    },
    body: JSON.stringify({
      "data": data,
      "type": type,
      "createdAt": createdAt
    }),
    url: `http://${elastic_search_url}/patients/_doc/${id}?pipeline=attachment`
  }

  return await request(requestOptions);
}

async function index_patient(patient_id) {
  let patient = await Patient.find(_id: patient_id);

  let data;

  for(let i = 0; i < patient.prescriptions.length; i++) {
    let fileContent = fs.readFileSync(patient.prescriptions[i])

    data = `${data} ${fileContent}`
  }

  if(!data) {
    return
  }

  createIndex(patient._id,
              new Buffer(data.trim()).toString('base64'),
              patient.type,
              patient.createdAt)
}

We need to call index_patient method on after save/update of patient model. In case we have existing patient data, index need to be created for all existing patients before initiating search.

Searching Data

Once the pipeline has been created and data has been indexed, we can search based on text in the attachment(prescriptions), type of patient or createdAt date range.

To make it a little more closer to reality, we will be adding limit and offset to data being fetched from elastic search to allow server side pagination to put less load on the server in case we have millions of patient records. We will also add a desc sort condition over createdAt date.

async function search(type, search, startDate, endDate, skip, limit) {
  let requestOptions = {
    method: 'POST',
    headers: 'Content-Type': 'application/json',
    body: JSON.stringify({
      "sort" [
        "createdAt": {
          "order": "desc"
        }
      ],
      "query": {
        "bool": {
          "must": [{
            "match": {
              "type": {
                "query": `*${type}*`
              }
            },
            "match": {
              "attachment.content": {
                "query": `*${search}*`,
                "operator": "and"
              }
            },
            "range": {
              "createdAt": {
                "gte": startDate,
                "lte": endDate
              }
            }
          }]
        }
      }
    }),
    url: `http://${elastic_search_url}/patients/_doc/_search?size=${limit}&from=${skip}`
  }

  return JSON.parse(await request(requestOptions))
}

let searchResult = await search(type, search, startDate, endDate, skip, limit)

ids = searchResult.hits.hits.map(r = r['_id'])

// Return this list of patients to the user
await Patient.find({
  _id: {
    $in: ids
  }
}).sort({
  'createdAt': 'desc'
})

The above search query will do a partial search in all the prescriptions of the patient along with range query for createdAt date, sort condition for createdAt, partial match for type and can be extended as per your use case.

Starting with a simple find query in mongo, we made use of power of elastic search to make complex search queries. What we saw is a very small subset of the actual power of elastic search. A lot more can be done using the combination of Elastic Search and any data store.
