Considering a GraphQL switch? Here's why it is worth it.

Deepak Prasanna

A tarkan

GraphQL is getting widely adopted across the tech industry, and for the uninitiated, it is an open-source query language and a runtime used for creating APIs. Before GraphQL, REST APIs were predominantly used for this purpose. It is here to stay and packs a punch with its new set of features.

We’ve helped a few businesses adopt GraphQL to their tech stack. In this article, we will discuss why you should consider adding GraphQL to your stack as well.

Is GraphQL just a buzzword?

Over the past decade, REST has become the standard for designing web APIs. Recently GraphQL has gained a lot of popularity due to its different approach and API design.

REST follows a structured way to design your endpoints. GraphQL follows a different approach in its design making the endpoints to be discoverable. The tooling and the eco-system for GraphQL are designed with this aspect in mind.

GraphQL makes use of a query language on the client-side which allows the client to be very specific about its data needs. On the server side, it comes with a runtime that allows functions called resolvers to fulfill the data request. This means that apps can load faster giving a better user experience.

Developers from the frontiers of both web and mobile are equally excited about GraphQL. If you have a web app, several packages are available for easier GraphQL integration. Similarly, SDKs are available to integrate GraphQL on Android and IOS platforms too.

Solutions given by GraphQL for the complex problems faced by developers are legitimately good. GraphQL also shines in the space where companies face scaling issues. Reputable companies like Github, Twitter, Paypal, etc have been using GraphQL in their production system for quite some time.

We feel that adding GraphQL to your stack is a significant decision. So let’s dive a bit deeper to cover some of the most compelling reasons to use GraphQL.

Reasons to use GraphQL

Strongly typed schema:

GraphQL exposes its API using a strongly-typed Schema. The Schema acts as a backbone for the entire GraphQL API.

The strongly-typed nature allows the developers to enjoy better tooling, editor support, and compile time error detection. This can save a significant amount of developer time which is valuable for every business.

Another advantage of strong-typing is that it allows the schema to be self documenting. Therefore it eliminates the need to perform additional documentation about the data attributes.

Data over-fetching:

The response of a REST API endpoint has a predefined structure. This fixed response structure forces the consumer of a REST endpoint to over-fetch additional data attributes, even when they are not needed. This can lead to increased bandwidth usage. Also, unnecessary CPU cycles are wasted in serializing and de-serializing the payload.

GraphQL solves this problem by allowing a client’s request to be expressive. Clients can construct a Query which can have a dynamic response structure. This allows clients to fetch data attributes that are exactly needed on a case-by-case basis.

Data under-fetching:

REST models its API endpoints as actions performed on a single resource. Following a RESTful convention will often lead to designing endpoints with a single domain entity. This causes data to be under-fetched. Clients have limited options when data from multiple resources are required to be fetched in a single request. The workaround is often to model endpoints as nested resources or creation of endpoints tailor-made for specific scenarios. This can create additional complexity and may lead to the creation of endpoints that are not RESTful.

In GraphQL, the domain entities and the operations are defined using GraphQL Schema Definition Language. It is very powerful but at the same time flexible enough to enable composition in a cleaner fashion.

Since GraphQL allows clients to compose queries, it eliminates the need to make multiple network calls. This leads to a better user experience avoiding the overhead involved in making multiple network calls.

Data Aggregation from different services(AKA BFF):

When your application starts growing, you might start relying on different services for different business needs. So your application would require calls to multiple data sources such as micro-services, 3rd party APIs, etc. to fulfill the data requests. The results from those data sources need to be aggregated and sent back to the client.

GraphQL allows you to combine results from multiple data sources and return a single response to the client, acting as a data-aggregator. We feel this is one of GraphQL’s biggest strengths.

Schema Stitching:

Schema Stitching is the process of combining schemas from two or more GraphQL APIs and exposing them as a single unified GraphQL API. According to Principled GraphQL, having a unified GraphQL API is key in order to maximize the value of GraphQL.

Your company should have one unified graph, instead of multiple graphs created by each team.

- Principled GraphQL

Giving each team enough independence for the development of their own graph is quite important. An organization with different teams can make use of this feature to combine smaller GraphQL APIs and expose it as a single larger GraphQL API to consumers.

Community:

GraphQL is open-source and the community around it is super active. Thanks to the GraphQL foundation, they are the driving force involved in keeping the GraphQL spec moving forward.

Companies like GitHub, Airbnb, Shopify, Paypal, and Trulia use GraphQL in their production system and they have seen impressive results with it.

As of today, GraphQL spec has been implemented in multiple programming languages. The options are plenty for GraphQL client implementations too, some of the popular ones are Relay, ApolloClient, urql and graphql-request.

The StackOverflow Trend for GraphQL is on the rise which shows the increase in people offering help for GraphQL-related questions.

Conclusion

GraphQL has shown some serious improvements over REST in data fetching, code splitting, developer tooling, and documentation. Since the GraphQL landscape is changing fast, we have faced instances where the resources found online can be out of date. It is good in a way that GraphQL is undergoing rapid transformation, and this will change over time as the system matures.

GraphQL has a lot more to offer, we’ve only touched upon the important aspects to consider. Like any new technology, even GraphQL has a bit of a learning curve. However, we feel it is totally worth it!
