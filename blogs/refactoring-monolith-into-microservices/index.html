Going for Service Mesh

One of the projects I was working on had a monolithic MEAN stack web application running on docker inside a virtual machine. We had multiple replicas for the backend for load balancing. After facing an outage due to a big batch of simulations API calls we had to update the architecture of the application to make it sustainable and scalable.

And our 6-month refactoring began, this was our chance to make some substantial changes we wanted in logging, caching, monitoring, etc.

Some highlights of the migration:

Baby steps

We listed down the microservices we want to separate out from the monolith according to priority and developed the one microservice maintaining the monolith as the point of entry.

This gave us the confidence that the functionality is not breaking even though we were doing significant changes in the architecture. After 12 sprints we have 13 microservices and the monolith was just an API gateway at this point.

Changed from Express to Fastify

We used the-benchmarker to find high-performance JavaScript frameworks and found that Fastify will be easy for the transition and faster in terms of API response.

the-benchmarker/web-frameworks

Need more than 2 commands to do something => ./Automate.sh

Automating processes like building new docker image versions, uploading to cloud image registry, applying changes to Kubernetes cluster, etc. helped us boost our productivity.

Custom logging for better monitoring

We developed a custom logging library that is compatible with Fastify and added customization according to the processing handled by the microservice. Logs were sent to logging microservice which indexes the logs in Elastic Search and accessed the logs using Kibana.

Elastic Search is open source, easy to spin up and configure, it can handle a large amount of data and index the data quickly. Plus we were already using Elastic Search for indexing and fast searching of entities.

Kibana can be used to perform a number of operations on the indexed data. We created visualizations and a dashboard in Kibana for monitoring.

Custom Caching and Purging library

The main intention of adapting microservices was to improve the performance of the application. So we used Redis, creating middleware for checking and sending cached result was the easy part, but finding out activities where cached result purges were the hard part.

Redis is an open-source in-memory data store. Redis can store key-value pairs. Again we were already using Redis for queuing-related operations.

Version Change and Package updates

As we were creating a new repository and project for every microservice we updated the Angular version, Node version, and deprecating packages as well.

Fix scalability & performance issues at the microservice level

We prioritized performance and scalability fixes, and while developing microservices we made the specific fixes which was much easier than fixing in monolithic.

Once the microservice is separated from monolithic, you have to deal with less code and while planning the development of microservice you can think of the fixes required, so you can manage major changes effectively.

Load testing script

Built custom load testing script which allowed us to send dynamic spikes to the cluster and dashboard allowed us to see the application status in real-time.

Going for Service Mesh

We used Istio which shipped tools like Grafana, Jaeger, Kiali, Prometheus, Zipkin, etc.
